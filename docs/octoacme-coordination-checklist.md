# OctoAcme Cross-Functional Coordination Checklist

## Purpose
This checklist ensures effective handoffs, feedback loops, and coordination between roles throughout the project lifecycle. Use it to maintain accountability and avoid gaps in communication.

---

## Project Initiation

### Initial Alignment
- [ ] **Project Manager** schedules kickoff meeting with all key stakeholders
- [ ] **Product Manager** shares initial problem statement and business context
- [ ] **Business Analyst** reviews preliminary requirements with stakeholders
- [ ] **Technical Lead** provides initial feasibility assessment
- [ ] **UX Designer** shares relevant user research or identifies research needs
- [ ] All roles review and align on success metrics

### Setup & Documentation
- [ ] **Project Manager** creates project charter and timeline
- [ ] **Scrum Master** sets up project board and defines team working agreements
- [ ] **Product Manager** creates initial product roadmap
- [ ] Roles and responsibilities documented using RACI matrix
- [ ] Communication plan established (standups, syncs, status updates)

---

## Requirements & Planning

### Requirements Gathering
- [ ] **Business Analyst** conducts stakeholder interviews and documents requirements
- [ ] **Product Manager** prioritizes requirements and defines acceptance criteria
- [ ] **UX Designer** conducts user research and validates user needs
- [ ] **Business Analyst** shares requirements documentation with team for review
- [ ] **Technical Lead** reviews requirements for technical feasibility
- [ ] **Developers** provide feedback on implementation complexity

### Design & Architecture
- [ ] **UX Designer** creates wireframes and shares with Product Manager for feedback
- [ ] **Technical Lead** creates technical design document
- [ ] **UX Designer** conducts design review with Developers and Technical Lead
- [ ] **Business Analyst** validates designs against business requirements
- [ ] **Product Manager** approves final designs
- [ ] Design and architecture artifacts linked in project documentation

### Sprint Planning
- [ ] **Scrum Master** schedules and facilitates sprint planning
- [ ] **Product Manager** ensures backlog is prioritized and user stories are ready
- [ ] **Business Analyst** confirms acceptance criteria are clear and testable
- [ ] **Technical Lead** identifies technical dependencies and risks
- [ ] **Developers** estimate work and commit to sprint capacity
- [ ] **UX Designer** ensures design specs are available for planned work
- [ ] Team agrees on sprint goal and definition of done

---

## Execution & Tracking

### Daily Coordination
- [ ] **Scrum Master** facilitates daily standup
- [ ] **Developers** report progress, plans, and blockers
- [ ] **Technical Lead** provides guidance on technical decisions
- [ ] **Scrum Master** tracks and escalates blockers
- [ ] **Project Manager** monitors critical path and dependencies

### Work In Progress
- [ ] **Developers** implement features according to acceptance criteria
- [ ] **Technical Lead** reviews code for quality and architectural consistency
- [ ] **UX Designer** reviews implementation for design fidelity
- [ ] **Business Analyst** validates implementation against business requirements
- [ ] **Developers** address review feedback and iterate

### Cross-Team Coordination
- [ ] **Project Manager** coordinates with dependent teams on handoffs
- [ ] **Technical Lead** aligns on integration points with other technical leads
- [ ] **Business Analyst** ensures data flows are validated across systems
- [ ] External dependencies tracked and communicated in status updates

### Quality Assurance
- [ ] **Developers** write and execute unit tests
- [ ] QA team validates against acceptance criteria
- [ ] **Business Analyst** performs user acceptance testing
- [ ] **UX Designer** validates user experience and accessibility
- [ ] **Technical Lead** reviews security and performance implications
- [ ] Bugs triaged and prioritized by Product Manager and Technical Lead

---

## Sprint Review & Feedback

### Demo Preparation
- [ ] **Scrum Master** schedules sprint review with stakeholders
- [ ] **Developers** prepare demo of completed features
- [ ] **Product Manager** invites relevant stakeholders
- [ ] **UX Designer** prepares to present design rationale if needed

### Review Session
- [ ] Team demonstrates completed work against acceptance criteria
- [ ] **Product Manager** validates features meet business goals
- [ ] **Business Analyst** confirms requirements are satisfied
- [ ] Stakeholders provide feedback and ask questions
- [ ] **Product Manager** accepts or rejects work based on Definition of Done
- [ ] Feedback captured and added to backlog for next sprint

---

## Release & Deployment

### Pre-Release Coordination
- [ ] **Project Manager** confirms release readiness with all stakeholders
- [ ] **Technical Lead** reviews deployment checklist and rollback plan
- [ ] **Developers** complete code freeze and final testing
- [ ] **Product Manager** reviews release notes for completeness
- [ ] **Business Analyst** validates business rules in staging environment
- [ ] **UX Designer** performs final usability check

### Deployment
- [ ] **Technical Lead** oversees deployment execution
- [ ] **Developers** execute deployment steps
- [ ] **Technical Lead** validates smoke tests pass
- [ ] **Project Manager** communicates release status to stakeholders
- [ ] On-call rotation and support plan activated

### Post-Release
- [ ] **Product Manager** monitors success metrics and user feedback
- [ ] **Technical Lead** monitors system health and error rates
- [ ] **Project Manager** conducts post-release review with team
- [ ] **Business Analyst** validates business outcomes align with expectations
- [ ] Issues triaged and prioritized for hotfixes if needed

---

## Retrospective & Continuous Improvement

### Retrospective Preparation
- [ ] **Scrum Master** schedules retrospective meeting
- [ ] Team members reflect on what went well and what to improve
- [ ] **Project Manager** reviews project metrics (velocity, cycle time, quality)
- [ ] Key incidents or challenges documented for discussion

### Retrospective Session
- [ ] **Scrum Master** facilitates retrospective using agreed format
- [ ] Team shares feedback in psychologically safe environment
- [ ] Successes celebrated and learnings captured
- [ ] Improvement opportunities identified and prioritized
- [ ] **Scrum Master** documents action items with owners and timelines

### Action Follow-Up
- [ ] **Scrum Master** tracks action items and reports progress
- [ ] **Project Manager** incorporates learnings into future project plans
- [ ] Process improvements implemented in next sprint
- [ ] **Technical Lead** ensures technical improvements are prioritized
- [ ] Retrospective notes shared with broader organization if relevant

---

## Handoff & Knowledge Transfer

### Documentation Handoff
- [ ] **Developers** update technical documentation and README files
- [ ] **Technical Lead** documents architecture decisions and patterns
- [ ] **Business Analyst** ensures requirements traceability is complete
- [ ] **UX Designer** updates design system and component library
- [ ] **Product Manager** creates product knowledge base entry
- [ ] **Project Manager** archives project artifacts in central repository

### Knowledge Sharing
- [ ] Team conducts knowledge transfer session if roles are changing
- [ ] Key decisions and rationale documented in decision log
- [ ] Contact points identified for post-project support
- [ ] Lessons learned shared with other teams via retrospective summary
- [ ] **Scrum Master** facilitates handoff ceremonies as needed

---

## Tips for Effective Coordination

### Communication Best Practices
- **Over-communicate changes:** When scope, timeline, or requirements change, notify all affected roles immediately
- **Document decisions:** Use decision logs to capture why choices were made
- **Single source of truth:** Keep project status, plans, and artifacts in one accessible location
- **Regular sync points:** Don't wait for issues to escalateâ€”maintain regular touchpoints

### Avoiding Common Pitfalls
- **Requirements ambiguity:** Ensure Business Analyst, Product Manager, and UX Designer align before development starts
- **Design-development disconnect:** UX Designer should be available during implementation for questions
- **Late blocker escalation:** Scrum Master and Project Manager should surface impediments early
- **Missing acceptance criteria:** Product Manager and Business Analyst must define clear criteria before sprint planning
- **Skipped code reviews:** Technical Lead must ensure code quality standards are maintained

### Accountability Reminders
- Each checklist item should have a clear owner (refer to RACI matrix)
- Track checklist completion in your project management tool
- Review checklist adherence during retrospectives
- Adapt checklist based on project size and complexity
